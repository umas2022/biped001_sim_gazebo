# 最小 LQR Demo 说明

这份文档对应：

[`examples/lqr_minimal_demo.py`](../examples/lqr_minimal_demo.py)

它现在同时承担两件事：

- 解释 LQR 最基本的概念
- 逐段解释这个最小示例代码

这样你只需要看这一份文档，就能把最小例子看明白，不需要再在多份文档之间来回跳。

## 1. 这个 demo 的定位

它不是正式控制器，也不是高保真物理模型。

它只是一个教学骨架，刻意只保留四步：

1. 定义离散状态空间模型
2. 定义 `Q` 和 `R`
3. 求出 LQR 增益 `K`
4. 用 `u = -Kx` 跑一个小仿真

它不依赖：

- ROS 2
- Gazebo
- 图形界面

所以更适合拿来理解控制主线。

## 2. LQR 最短解释

LQR 是一种状态反馈控制方法。

它依赖的不是神经网络模型，而是一个数学模型：

`x[k+1] = A x[k] + B u[k]`

这里：

- `x`
  是状态向量
- `u`
  是控制输入
- `A`
  描述系统自己会怎么演化
- `B`
  描述控制输入会怎样影响系统

LQR 最终求出一个反馈增益：

`u = -Kx`

也就是根据完整状态 `x` 来决定控制输入，而不是只盯着单个误差项。

## 3. 它和 PID 的区别

PID 更像是围绕一个误差做反馈：

`u = Kp * e + Ki * ∫e dt + Kd * de/dt`

LQR 更像是：

- 先写出整个系统的状态模型
- 再用一个统一的代价函数权衡状态误差和控制输出
- 最后得到状态反馈增益 `K`

所以对两轮平衡车这种多状态耦合系统，LQR 往往比单环 PID 更自然。

## 4. 这个最小例子的状态是什么

示例里只用两个状态：

`x = [theta, theta_dot]`

也就是：

- `theta`
  倒立摆相对竖直方向的倾角
- `theta_dot`
  倾角变化速度

这比正式项目简单很多。

正式项目里使用的是四状态：

- `wheel_pos_error`
- `wheel_vel`
- `pitch_error`
- `pitch_rate`

所以这个 demo 可以看成：

“先学最简单的两状态倒立摆 LQR，再回去看工程版四状态控制器。”

## 5. 把每个符号都放回倒立摆这个物理对象里看

如果只看公式，`x`、`u`、`A`、`B`、`Q`、`R` 很容易显得抽象。

放回这个最小倒立摆例子里，它们可以这样理解：

- `x = [theta, theta_dot]`
  当前倒立摆的状态。
- `theta`
  杆偏离竖直方向的角度。
- `theta_dot`
  这个倾角变化得有多快。
- `u`
  控制器施加的纠正作用强度。

这里的 `u` 在教学模型里是抽象输入，不必过早把它死理解成某一种具体硬件量。

你可以先把它理解成：

“一种能够改变倒立摆角速度，从而把杆拉回竖直附近的控制作用。”

在真实系统里，这个量可能最终对应：

- 电机力矩
- 轮子驱动力
- 轮速命令

但在这个最小例子里，它只需要承担一个角色：

“作为控制输入，影响倒立摆接下来怎么动。”

同样地：

- `A`
  描述如果完全不控制，倒立摆自己会怎么演化。
- `B`
  描述控制输入 `u` 会怎样影响倒立摆状态。
- `Q`
  描述我们有多在意状态偏差。
- `R`
  描述我们有多在意控制动作不要太猛。

## 6. 代码结构非常简单

文件里只有两个主要函数：

1. `solve_lqr_gain(a, b, q, r)`
   根据 `A/B/Q/R` 求出 `K`
2. `main()`
   定义模型、设置初值、跑仿真并打印结果

主线可以压缩成：

```python
a, b = ...
q, r = ...
k = solve_lqr_gain(a, b, q, r)

x = initial_state
for each step:
    u = -Kx
    x = A x + B u
```

## 7. `solve_lqr_gain(...)` 在做什么

这个函数对应的离散系统是：

`x[k+1] = A x[k] + B u[k]`

LQR 想最小化长期总代价：

`sum(x^T Q x + u^T R u)`

含义很直接：

- 状态偏差太大，要罚
- 控制输出太大，也要罚

在这个 2 状态例子里，`x` 是：

`x = [theta, theta_dot]`

而 `u` 是一个标量，所以这个代价函数可以直观理解成：

“如果杆歪得厉害，要付代价；如果杆晃得很快，也要付代价；如果为了救摆打得太猛，同样要付代价。”

由于这里：

```python
q = np.diag([40.0, 4.0])
r = np.array([[0.5]])
```

所以它大致等价于长期累计：

`40 * theta^2 + 4 * theta_dot^2 + 0.5 * u^2`

这就把抽象公式翻译成了很直观的物理偏好：

- 不希望杆偏离竖直太多
- 也不希望杆大幅摆动
- 还不希望控制器输出过于暴力

### 这个式子是“怎么来的”

这不是某一条唯一物理定律直接推出来的，而是控制设计者主动选定的一种目标函数。

设计思路是：

- 我希望系统稳定回零
- 但我又不希望控制输入无限大

于是就把这两个目标写成一个统一的二次型代价：

`sum(x^T Q x + u^T R u)`

LQR 的理论结论是：

对于线性系统

`x[k+1] = A x[k] + B u[k]`

和这种二次型代价函数，最优控制律可以写成状态反馈形式：

`u = -Kx`

函数内部先用 `solve_discrete_are(...)` 求 Riccati 方程的解 `P`，再由 `P` 计算最优反馈增益 `K`。

初学阶段不需要先背推导，先记住这条链路就够了：

`A/B` 定义系统，`Q/R` 定义偏好，LQR 据此求出 `K`。

## 8. 程序里的具体数值，对应倒立摆的什么含义

这份最小示例里，最值得建立物理直觉的是下面这些数值。

### `dt = 0.02`

表示系统每 `0.02 s` 更新一次，也就是：

- 20 ms 一拍
- 50 Hz

物理上可以理解成：

“控制器每隔 20 ms 观察一次当前姿态，并重新计算一次纠正动作。”

### `9.0`

它出现在：

```python
[9.0 * dt, 1.0]
```

里。

这个数表示一种简化后的“不稳定强度”。

含义是：

- 只要杆已经偏了
- 它下一拍就会继续朝着更倒的方向发展

这个数越大，表示倒立摆越“不老实”，越需要更积极的控制才能压住。

### `1.0` 在 `B` 里

它出现在：

```python
b = np.array([
    [0.0],
    [1.0 * dt],
])
```

里。

表示控制输入 `u` 对角速度变化的作用强度。

直觉上就是：

- 同样大小的 `u`
- 到底能多大程度推着这个摆的运动方向发生改变

### `Q = diag([40.0, 4.0])`

这表示在这个例子里：

- 比起“晃得快不快”
- 更优先惩罚“姿态偏没偏离竖直”

因为：

- `theta` 的权重是 `40`
- `theta_dot` 的权重是 `4`

也就是说，控制器眼里“歪了”比“晃得快”更严重。

### `R = 0.5`

表示控制输入也有代价，但当前设置并不算特别大。

这意味着：

- 控制器愿意比较积极地出手
- 但仍然不希望无限放大控制动作

### `x = [0.20, 0.0]`

表示初始状态是：

- 倾角 `0.20 rad`
- 角速度 `0`

`0.20 rad` 大约等于 `11.46 deg`。

物理上就是：

- 一开始杆已经歪了十几度
- 但还没有继续快速摆动

这是一个很适合教学的起点，因为它足够简单：

- 有明显误差
- 但不混入额外初速度

### `u` 限幅到 `[-5, 5]`

程序里：

```python
u = max(-5.0, min(5.0, u))
```

表示控制器虽然可以算出一个理想输出，但最终只能施加有限大小的纠正作用。

物理上很好理解：

- 真实执行器能力有限
- 不可能输出无限大力矩或无限大驱动力

这一步不是 LQR 理论核心，而是很常见的工程保护。

## 9. `main()` 里每一步的控制意义

### `dt = 0.02`

表示离散仿真每 20 ms 更新一次，也就是 50 Hz。

### `A`

```python
a = np.array([
    [1.0, dt],
    [9.0 * dt, 1.0],
])
```

它表达的是：

- 当前角度会受到当前角速度影响
- 当前倾角会继续推动系统往“更倒”的方向发展

这个写法是简化教学模型，不是完整实体推导。

### `B`

```python
b = np.array([
    [0.0],
    [1.0 * dt],
])
```

表示控制输入 `u` 先影响角速度，再通过角速度影响角度。

### `Q`

```python
q = np.diag([40.0, 4.0])
```

表示更重视把 `theta` 压小，其次才是 `theta_dot`。

### `R`

```python
r = np.array([[0.5]])
```

表示不希望控制输出过于激进。

粗略直觉是：

- `Q` 大一些，控制通常更积极
- `R` 大一些，控制通常更保守

## 10. `u = -Kx` 到底怎么落地

仿真循环里最核心的一行是：

```python
u = -(k @ x).item()
```

它表示：

- 当前状态偏了多少
- 当前状态变化有多快
- 这些量一起经过增益矩阵 `K`
- 生成一个负反馈控制量 `u`

前面的负号就是“往回拉”的意思。

如果状态偏到正方向，控制要往负方向拉；反过来也一样。

## 11. 限幅和状态更新

后面两步是：

```python
u = max(-5.0, min(5.0, u))
x = a @ x + b.flatten() * u
```

第一行是工程上常见的输出限幅。

第二行就是把离散模型真正跑起来：

- `A x`
  系统自己的演化
- `B u`
  控制输入带来的变化

## 12. 终端输出应该怎么看

脚本会打印：

- `A`
- `B`
- `Q`
- `R`
- `K`
- 随时间变化的 `theta`、`theta_dot`、`u`

看结果时重点观察三件事：

1. `theta` 是否持续接近 `0`
2. `theta_dot` 是否也逐渐接近 `0`
3. `u` 是否在合理范围内逐步减小

如果这三件事同时发生，说明闭环在收敛。

## 13. 这个例子的简化边界

这个 demo 适合回答：

- LQR 的最小闭环长什么样
- 程序里怎么从 `A/B/Q/R` 走到 `u = -Kx`

它不适合回答：

- 真正机器人为什么一定这样建模
- 为什么这个参数能直接上真机
- 为什么加入饱和后仍然严格最优

因为它本来就只是教学版最小例子。

## 14. 从最小 demo 过渡到正式项目

回到正式控制器时，你可以这样建立映射：

- 这里的 `x`
  对应工程版更完整的状态向量

  对应工程版轮子控制输出
- 这里的 `A/B`
  对应工程版的线性近似模型
- 这里的 `Q/R`
  对应工程版调参偏好
- 这里的离散循环
  对应工程版每个控制周期的读状态、算输出、发命令

骨架没有变，只是工程版状态更多、保护更多、链路更多。
